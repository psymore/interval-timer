# Production Readiness Review — interval-timer

Audit date: 2026-07-08. Read-only audit — no application source code was modified. Full detail behind every summarized finding lives in the companion reports in this same `production-review/` directory: `electron-security-review.md`, `code-quality-review.md`, `semgrep-report.txt`, `trivy-report.txt`, `gitleaks-report.txt`, `codeql-report.txt`, `knip-report.txt`, `madge-report.txt`, `biome-report.txt`, `npm-audit-report.txt`.

## Executive Summary

The app is well-engineered for its size — consistent architectural layering (`js/logic/` state machines, `js/views/` render functions, `js/alarm/` strategy+factory pattern), no `child_process` usage anywhere, a real Content-Security-Policy on both windows, a minimal and correctly-scoped `preload.cjs` with no raw `ipcRenderer` exposure, and the Spotify client secret is genuinely confined to the main process. Dependency hygiene is clean: **0 vulnerabilities** (npm audit, Trivy), **0 leaked secrets in git history** (Gitleaks), and only **1 truly unused export** in the whole renderer (Knip, once correctly configured with real entry points — see note below).

The one finding that gates the "ready for production" verdict is a **CRITICAL, concretely-exploitable local path-traversal / arbitrary-file-read vulnerability** in the local HTTP server (`main.js:74-98`) that the renderer is served from. It was found independently by both Semgrep and a manual Electron-security audit, and its exploitability was verified by hand (a crafted request to the app's fixed, predictable port `127.0.0.1:47821` escapes the app directory and reads arbitrary files off the user's disk). It is a small, well-understood fix — but it must be patched and verified before this app should be distributed publicly, which per project context is the near-term plan for this repository.

Beyond that, there are real but non-blocking reliability gaps (no recovery if the renderer crashes or the initial page load fails — the app would just show a permanently blank window), some maintainability drag (`main.js` has grown into a 707-line file mixing five unrelated concerns), and a CSS specificity war plus token drift between the main and mini windows' stylesheets. None of these block shipping on their own.

One process note: during this audit, running Biome's formatter check with no scope configuration caused it to read and print the real Spotify client secret (from the gitignored, locally-only `spotify-credentials.json`) into a report file. This was caught and redacted before it left the local filesystem (see **Finding-P1** below) — but it's a useful, generalizable lesson: `.gitignore` only protects `git`; every other tool that walks the filesystem needs its own exclusion list.

## Findings

Severity counts: **1 CRITICAL · 6 HIGH · 11 MEDIUM · 9 LOW · 8 SUGGESTION** (34 total, across security + code quality + tooling; excludes purely-clean tool runs).

### CRITICAL

| # | Finding | File(s) | Why it matters | Fix |
|---|---|---|---|---|
| 1 | **Path traversal / arbitrary file read** in the local static-file server. `req.url` is URL-decoded and `path.join`'d onto `__dirname` with no containment check; `path.join` does not stop `..` from escaping the base directory. Verified: a request for `/../../../../../../Windows/win.ini` resolves to `C:\Windows\win.ini` and is read and returned. The server binds to a **fixed, hardcoded port** (`127.0.0.1:47821`, chosen deliberately so `localStorage` persists across restarts), so this is reachable — without any port-discovery step — by any other local process running as the same OS user, or by a web page open in the user's regular browser (via `<script src>`/`<img>`/`fetch`, independent of Same-Origin restrictions on the request itself). | `main.js:74-98` (`handleLocalServerRequest`) | Full unauthenticated local arbitrary-file-read primitive, reachable by anything on the machine, at all times the app is running, with a fixed and publicly-documented (in this very repo's own comments) port. | Resolve with `path.resolve`/`path.normalize`, then verify the result still starts with `__dirname` (`path.sep`-safe) before `fs.readFile`; reject otherwise. Better: serve only from a fixed allow-list (`index.html`, `mini.html`, `js/**`, `css/**`, `assets/**`) rather than resolving arbitrary request paths at all. |

### HIGH

| # | Finding | File(s) | Why it matters | Fix |
|---|---|---|---|---|
| 2 | `/local-audio/` route reads **any absolute file path** with an allowed extension (`.mp3/.wav/.ogg`), no directory containment at all — a narrower but still real variant of Finding 1. | `main.js:52-72` | Same reachability profile as Finding 1 (fixed port, no auth); an attacker who knows or guesses a path can exfiltrate the contents of any matching-extension file on disk. | Validate the resolved path against a main-process-side allowlist populated only by the user's own file-dialog selection, or restrict to a fixed user-data subdirectory. |
| 3 | Spotify **access and refresh tokens stored in plaintext `localStorage`** instead of Electron's `safeStorage` (OS-keychain-backed encryption). The refresh token is long-lived and doesn't expire on its own. | `js/alarm/AlarmManager.js:288-294`, `js/alarmModal.js:247-249` | Anything with filesystem access to the app's profile directory (or a future XSS/DevTools exposure) can read a durable-access Spotify credential. Client secret handling is correctly done main-process-only elsewhere — this is the one place that pattern wasn't extended to the token itself. | Move token storage to the main process via IPC; encrypt at rest with `safeStorage.encryptString`/`decryptString` (Windows DPAPI-backed). |
| 4 | `main.js` has grown into a **707-line God-file** mixing 5+ unrelated concerns: the local HTTP server, main/mini window + tray lifecycle, app lifecycle wiring, preset/file-picker IPC, and a complete Spotify OAuth flow (auth window + token exchange/refresh). | `main.js` (whole file) | This is the one file every future feature touches, and the one place in the codebase that breaks the separation-of-concerns discipline applied everywhere else (`js/alarm/*`, `js/logic/*`, `js/views/*`). Concentrates merge conflicts and regression risk. | Extract `lib/localServer.js`, `lib/windows.js` (main+mini+tray), `lib/presetsIpc.js`, `lib/spotifyAuth.js`; `main.js` becomes wiring only. |
| 5 | **No error handling around the main window's initial page load.** `mainWindow.loadURL(...)` has no `.catch()` and no `did-fail-load` listener (contrast: the Spotify auth window *does* have one). | `main.js:251` | If the local server isn't up yet, or loopback is blocked by a firewall/AV, the user gets a permanently blank window with zero feedback and a silently-swallowed rejected promise. | Add a `did-fail-load` handler (retry once / show an in-window error) and `.catch()` the load promise to log the failure. |
| 6 | **No renderer crash/hang recovery** — no `render-process-gone` or `unresponsive` listener anywhere. A renderer crash (plausible given YouTube IFrame API + Spotify `fetch` + `MutationObserver` usage) leaves a dead window with no reload path; the tray's "Open" just refocuses it. | `main.js` (absent) | Same class of problem as #5 from a different trigger — a "blank/frozen app" is the single worst first impression a production desktop app can give. | Listen for `render-process-gone`/`unresponsive`; offer `mainWindow.reload()`, ideally behind a dialog if a timer is running so it isn't silently reset. |
| 7 | `app.on("activate", async () => { ... await createWindow(); })` is its own independent async function whose rejections are **not** covered by the startup chain's `.catch()`. | `main.js:391-393` | A load failure triggered by a later `activate` (e.g. from the tray, on macOS-style re-open) becomes a second, completely unlogged unhandled rejection. | Wrap the body in its own `try/catch` matching the top-level pattern. |

### MEDIUM

| # | Finding | File(s) | Why it matters | Fix |
|---|---|---|---|---|
| 8 | No `session.setPermissionRequestHandler` — permission requests from embedded remote content (YouTube iframe, Spotify origins) fall back to Electron's default rather than an explicit app policy. | `main.js` (absent) | Cheap defense-in-depth; combined with the app's global `autoplayPolicy: "no-user-gesture-required"`, there's currently no backstop if embedded content ever prompts for camera/mic/geolocation/notifications. | Add `session.defaultSession.setPermissionRequestHandler((wc, permission, cb) => cb(false))` (deny-all; nothing in this app needs any browser permission today). |

### Process note (not a code finding)

**P1 — Audit tooling read and printed a real secret into a report file, redacted before any external exposure.** Running `biome check .` with no scope configuration caused it to read `spotify-credentials.json` (gitignored, never committed, holds the real Spotify client id/secret) and print the actual secret value into `production-review/biome-report.txt` as part of a formatting diff. This was caught immediately: the secret value was grepped for and confirmed absent from every other report file, the leaked lines were redacted in place, and a scoped `biome.json` (`files.includes`: only `main.js`, `preload.cjs`, `js/**/*.js`, `*.html`, `css/**/*.css`) was added so the re-run — and all future runs — never touches the credentials file at all. **Lesson for this project (and generally): `.gitignore` only stops `git` from tracking a file; every other tool that walks the filesystem (linters, formatters, static analyzers) needs its own explicit exclusion, or it will happily read and can leak secrets that are otherwise "safe" by git convention.**

## Tool installation & bundling verification

`semgrep` (pip), `trivy`/`gitleaks` (standalone binaries, downloaded outside the repository entirely) were installed as local dev-only tooling, never added to `package.json`. `knip`, `madge`, and `@biomejs/biome` were added to `package.json` **`devDependencies`** only (confirmed: `dependencies` still lists only `electron-store`). No `build.files` override exists, so electron-builder's default packaging applies — which by design resolves and bundles only the production dependency tree, excluding `devDependencies` node_modules. Combined with the standalone binaries living entirely outside the project directory, **none of the tools installed for this audit can end up in the production bundle.** (Verified statically via `package.json` inspection and electron-builder's documented default behavior; a full `npm run dist` rebuild was not run as part of this verification since it wasn't necessary to establish the guarantee and would have produced an installer artifact not requested by this audit.)

## Conclusion

**❌ NOT READY FOR PRODUCTION**

The gate is a single, well-understood, quickly-fixable issue: the CRITICAL local path-traversal / arbitrary-file-read in the local HTTP server (`main.js:74-98`, Finding #1), corroborated independently by both static analysis and manual verification. Everything else found here — including the HIGH-severity plaintext token storage and the reliability gaps around crash/load-failure recovery — is real but does not, on its own, warrant blocking a release; they should be scheduled as prompt follow-up work.

**Recommended before shipping v1.0.0:**
1. Fix Finding #1 (path traversal) and Finding #2 (`/local-audio/` unconstrained absolute-path read) together — both live in the same ~50 lines of `main.js` and share the same fix shape (path containment / allowlist).
2. Fix Finding #3 (Spotify tokens in plaintext `localStorage`) before or shortly after — same trust boundary the client-secret handling already gets right elsewhere in this app.
3. Everything else in this report (HIGH reliability gaps, MEDIUM/LOW/SUGGESTION items) is appropriate as fast-follow work, not a release blocker.
